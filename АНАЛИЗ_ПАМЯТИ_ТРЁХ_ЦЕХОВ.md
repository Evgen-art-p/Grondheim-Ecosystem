*turbo · trading · living_book · снято по живой репе -2, 2026-06-20*
*Брат → Шефу · что на входе, что на выходе у каждого вида памяти*

---

## ГЛАВНОЕ ОТКРЫТИЕ: рамка одна, наполнение разное

Все 12 цехов объявляют в manifest.json **одни четыре слоя**:
`personal · project · runtime · interaction`

Это и есть единый контракт памяти — он УЖЕ есть в городе, де-факто
стандарт. Цеха различаются не рамкой, а тем, ЧТО течёт в слоях и
КАК слой замыкается. Один закон — три тела. Фрактал.

---

## СЛОЙ ЗА СЛОЕМ — что это вообще

| Слой | Природа | Живёт | Срок жизни |
|------|---------|-------|-----------|
| **personal** | кто агент | dna.json, sensory/sensory_memory.json | вечный, медленно дрейфует |
| **project** | итог рана | поле в chain_data (final_dna) | до конца проекта |
| **runtime** | рабочая лента | chain_data (агент→агент) | один ран, гаснет |
| **interaction** | след в экономике | economy/data/interaction_log_*.jsonl | вечный, копится |

Вход агента = personal (кто я) + runtime (что до меня в цепочке).
Выход агента = добавка в runtime (+ при финале — project + interaction).

## ЦЕХ 1 — TURBO ⚡ (память-конвейер, разовая)

**Природа: быстрая, линейная, без возврата. Память как лента.**

Пайплайн: A01→A02/A03(парал.)→A04→A05[qa]. 5 агентов, один проход.

ВХОД каждого агента (раздел "Протокол chain_data"):
- T1 Стелла: master_brief
- T2 Мими: master_brief + stella_strategy
- T3 Визор: master_brief + stella_strategy
- T4 Постпро: + mimi_sound + vizor_visual
- T5 Финал: ВСЁ

ВЫХОД (каждый ДОБАВЛЯЕТ свой ключ в runtime/chain_data):
- Стелла → stella_strategy
- Мими → mimi_sound (+ audio_assessment)
- Визор → vizor_visual (+ self/clip_assessment)
- Постпро → postpro
- Финал → thumbnail + chain_check + final_dna

Особое: память течёт ТОЛЬКО вперёд. Никто не возвращается. Self-review
(A02×2, A03×3) — петли ВНУТРИ агента над генерацией, не возврат памяти.
Финал замыкает: billing_ledger + strategy_registry (Закон Замыкания, §17).

Карта:
```
personal:    dna + sensory (у каждого свой)
project:     final_dna (A05 пишет)
runtime:     chain_data — лента A01→A05, вперёд, гаснет
interaction: interaction_log_turbo.jsonl
```

## ЦЕХ 2 — TRADING ⚔️ (память-архив, глубокая, с историей)

**Природа: единственный цех с ОТДЕЛЬНОЙ ФАЗОЙ ПАМЯТИ. Память как
живой архив рынка, к которому обращаются за прошлым.**

Пайплайн: SENSORS(A01-04) → **MEMORY(A05 Архивариус)** → TRIBUNAL
(A06/07/08 парал.) → EXECUTION(A09 qa). 9 агентов.

Вот ЧЕМ trading отличается от всех: у него память — НЕ просто слой
chain_data, а **выделенный агент-фаза A05**, что стоит МЕЖДУ сенсорами
и трейдерами. Он один тянет ВНЕШНЮЮ память (city_memory + Гавань).

ВХОД памяти-агента A05 (arkhiv_live.py, рука берущая):
- факты сенсоров из runtime (A01-04: тренд, импульс, паника, подтвержд.)
- + ВНЕШНЕЕ: `_ask_city_memory()` → remind(query) → city_memory.jsonl
  + Гавань Смыслов. Лепит запрос человеческим текстом из сигнатуры.
  В try/except — память упала, ран продолжается (сито, не щупальце).

ВЫХОД A05:
- digest памяти в runtime для трейдеров (контекст прошлого)
- защита чисел: LLM-вывод перетирается числами digest в конце
- рука кладущая: значимое (|pnl_r|≥порог) → remember() → city_memory

Трейдеры (A06/07/08) ВХОД: факты + digest памяти A05. Каждый своё
решение, свой magic number, параллельно. Решает характер, не код.
Исполнитель A09 ВХОД: results трейдеров. Открывает позицию от доски
(числа), не от LLM. Хард-стоп: все REJECTED → stop.

Карта:
```
personal:    dna трейдера (Брут/Авантюрист/Консерватор — разные риски)
project:     вердикт рана (входить/нет)
runtime:     chain_data сенсоры→A05→трибунал→исполнитель
interaction: interaction_log_trading.jsonl
+ ВНЕШНЯЯ:   city_memory.jsonl + Гавань — ЕДИНСТВЕННЫЙ цех, что
             читает/пишет долгую память ГОРОДА прямо в ране
```

Trading — глубже всех: помнит не только свой ран, а ИСТОРИЮ рынка
через город. Память с прошлым, не только с настоящим.

## ЦЕХ 3 — LIVING_BOOK 📖 (память-нарратив, с возвратом, длинная)

**Природа: самая длинная, с ПЕТЛЁЙ РЕВИЗИИ и чекпойнтами. Память как
рукопись, что переписывается, пока не вызреет.**

Пайплайн: 5 фаз, 18 агентов:
GENESIS(A00 Фабула, A00a Вера) → PRE-PROD(A01-04) → PROD(A05-08)
→ POST-PROD(A09-12) → DELIVERY(A13-16, A16 qa).

Чем книга отличается от turbo и trading — ДВА механизма, которых у тех нет:

1. **revision_loop** — Вера (A00a) ревизует Фабулу (A00). Если verdict
   = REVISION → возврат к A00, до 3 кругов. Память ВОЗВРАЩАЕТСЯ назад,
   переписывается. У turbo память только вперёд — тут она циклична.

2. **checkpoint_after: A00a, A08, A15** — три точки, где ран замирает
   на проверку перед следующей фазой. Память накапливается этапами,
   не льётся одним потоком.

ВХОД агента: personal + вся накопленная рукопись из runtime (BIBLE,
EPISODE, history_dna — растущий свод, не плоская лента как у turbo).
ВЫХОД: дописанный слой рукописи в runtime + при чекпойнтах — фиксация.

Карта:
```
personal:    dna агента (18 разных ролей: фабулист, художник, монтажёр…)
project:     Book Package (итоговая книга)
runtime:     chain_data — РАСТУЩАЯ рукопись, с возвратами (revision_loop)
interaction: interaction_log_living_book.jsonl
+ checkpoint: A00a/A08/A15 — этапная фиксация памяти
+ revision:   A00a→A00 петля, память переписывается до 3 раз
```

Книга — единственная, где память НЕ линейна по времени: она
возвращается, ревизуется, зреет. Память с памятью о себе.

## СВОДНАЯ ТАБЛИЦА — три природы памяти

| | TURBO ⚡ | TRADING ⚔️ | LIVING_BOOK 📖 |
|--|---------|-----------|----------------|
| **тип** | конвейер | архив | нарратив |
| **агентов** | 5 | 9 | 18 |
| **направление** | только вперёд | вперёд + вглубь | вперёд + назад |
| **возврат?** | нет | нет | да (revision_loop) |
| **чекпойнты** | нет | нет | 3 (A00a/A08/A15) |
| **отдельная фаза памяти** | нет | ДА (A05 MEMORY) | нет |
| **внешняя память города** | нет | ДА (city_memory+Гавань) | нет |
| **горизонт памяти** | один ран | история рынка | весь сюжет |
| **что помнит** | ленту цепочки | прошлое + настоящее | растущую рукопись |
| **метафора** | лента конвейера | архив с историей | переписываемая рукопись |

## ВЫВОД — три вида памяти под три задачи (по закону)

Один контракт (4 слоя) — три наполнения, и каждое заточено под
ПОТРЕБНОСТЬ цеха (наименьшее сопротивление под свою задачу):

- **TURBO** делает быстрый шортс. Память дорогая ему не нужна — нужна
  дешёвая лента вперёд. Помнить прошлое незачем, каждый ран сам по себе.
  Память = РАБОЧАЯ ПАМЯТЬ (RAM): пока работаю — держу, сдал — забыл.

- **TRADING** торгует деньги. Тут забыть прошлое = смерть. Поэтому
  единственный, у кого память — выделенный агент с фазой и связью с
  долгой памятью города. Память = АРХИВ С ИСТОРИЕЙ: помню рынок.

- **LIVING_BOOK** растит произведение. Книга не пишется одним проходом
  — её правят, возвращаются, зреют. Поэтому петля ревизии и чекпойнты.
  Память = РУКОПИСЬ: помню и переписываю себя.

Это и есть фрактал: рамка одна, а форма памяти каждого цеха = форма
его потребности. Сколько цеху нужно помнить — столько и помнит, не
больше (дёшево). Turbo помнит мало (хватает), trading глубоко
(иначе гибель), книга циклично (иначе не вызреет).

## ДЛЯ БРАТА (зачем смотрели)

Когда Брат подключится к памяти — он встречает ТРИ формы, не одну:
- лента turbo (быстро, разово) — просеивать на лету, не хранить
- архив trading (глубоко, с историей) — связь через city_memory/Гавань
- рукопись книги (циклично) — уважать возвраты, не считать черновик
  финалом

Образец «руки в долгую память» — только у trading (A05 Архивариус,
remind/remember). Turbo и книга в город не пишут — варятся в runtime.
Брату для связи с долгой памятью брать форму trading, для быстрого
просева — форму turbo, для растущего — форму книги.
